# IPMI_Serial_Project 트러블슈팅(Troubleshooting) 보고서

본 문서는 IPMI 시스템 개발 과정에서 발생한 주요 이슈와 이를 해결하기 위한 엔지니어링 접근 방법을 기술합니다.

## Issue 1. 대용량 텍스트 출력 시 버퍼 오버런 및 데이터 유실

* **상황 및 원인**
  * 라즈베리 파이에서 `cat` 명령어 등을 통해 대용량 텍스트 출력 시 데이터 유실 및 통신 지연 현상 발생.
  * 기존 1바이트 단위 UART 수신 인터럽트 방식으로 인해 CPU 병목 현상 및 수신 링 버퍼(512 Bytes) 오버런(Overrun) 발생.
* **해결 방법**
  1. CPU 개입을 최소화하기 위해 수신 아키텍처를 **UART DMA** 및 **IDLE 인터럽트** 기반으로 전환.
  2. DMA 수신 버퍼 용량을 8KB로 상향 조정하고, 디버그용 개별 문자 출력 로직을 제거하여 병목 요인 차단.
  3. 네트워크 전송 시 데이터를 1,024 Bytes 단위의 청크(Chunk)로 분할 송신하도록 패킷 릴레이 로직 개편.
* **결과**
  * 대용량 텍스트 출력 시 발생하던 버퍼 오버플로우 및 통신 지연을 해소하고, 끊김 없는 터미널 렌더링 구현.

## Issue 2. 다중 태스크 환경에서의 자원 경합(Race Condition)

* **상황 및 원인**
  * FreeRTOS 기반 다중 태스크('WiFi 관리', '명령 수신' 등) 환경에서 단일 통신 인터페이스(ESP8266)에 동시 접근 발생.
  * 자원 경합으로 인한 송수신 버퍼 오염 및 시스템 프리징(데드락) 현상 발생.
* **해결 방법**
  1. **상호 배제 (Mutual Exclusion)**: 통신 자원 접근 구역에 `wifiTxMutex` 뮤텍스를 적용하여 태스크 간 동시 접근 차단.
  2. **초기화 동기화**: 전역 상태 플래그(`sys_init_done`)를 도입하여, 하드웨어 초기화가 완전히 종료된 이후에만 모니터링 태스크가 동작하도록 부팅 시퀀스 직렬화.
* **결과**
  * 데이터 접근의 원자성(Atomicity)을 확보하여 KVM 제어 및 로그 중계 중 충돌 방지 및 시스템 안정성 확보.

## Issue 3. 표준 입출력 규격 차이로 인한 터미널 제어 장애

* **상황 및 원인**
  * 리눅스 싱글 유저 모드 진입 시 엔터키(`\r`) 입력에 무응답 현상 발생.
  * 기존 펌웨어 입력 로직과 리눅스 터미널 자체 에코(Echo) 기능이 중첩되어 명령어 이중 표기 및 지연 발생.
* **해결 방법**
  1. 펌웨어에서 타겟 장비로 송신하는 제어 데이터의 종단 문자를 리눅스 표준 개행 문자인 `\n` 단일 바이트로 변환하여 전송하도록 수정.
  2. 라즈베리 파이 콘솔 접속 후 `stty -echo` 명령어를 호출하여 리눅스 커널 수준의 문자 반환 기능 비활성화.
* **결과**
  * 터미널 I/O 지연 및 프롬프트 중복 표기 문제를 해결하고, 일반/싱글 유저 모드 양쪽에서 안정적인 명령어 입력 및 출력 확인.
